Identifying Counts Using the DAG inside SparkUI - Apache Spark | Databricks

Recently I've come across a curious situation: While printing two counts that were equal right before an if statement, the if statement comparing both of these counts was returning false. How can that be if both counts were printing the same value on the console right before going in?

After diagnosing them better using the SparkUI, I've figured out that inside the if statement, the counts were different. So, if you need to understand (and debug) your counts, you don't need to keep printing them (because they can lead to misunderstanding), you can find them inside a DAG visualization inside SparkUI

How does Spark evaluate a count?

The first thing you need to understand is that Spark is lazy. Every time you run a df.count() it triggers an action, which leads to a job. A .count() evaluates the entire dataframe every time it runs. It does not matter if the count that you print says 1 because triggering the df.count() again can give you 2 right after it. Confusing right?

This can happen when the dataframe you're operating has data from underlying queries on tables that can be updated between df.count() runs. Since Spark evaluates the entire dataframe logical plan every df.count() statement, it will go back and read the underlying data every time of that data frame and count it.

Take a look at the example below:

df1.count()
df2.count()

if( df1.count() == df2.count()):
    print("count matches, proceeding further")
else:
    print("count does not match!")

Let's evaluate this code:

df1.count() # returns 100
df2.count() # also returns 100

if( df1.count() == df2.count()): # should return true right? But it was returning false some of the times
    print("count matches, proceeding further")
else:
    print("count does not match!")

What happens, in sequence is this:

  1. Spark triggers a job (or multiple ones) for the first df1.count() function, which heads back to the data frame itself and recompiles it.
  2. It also triggers another job for the second df2.count() operation.
  3. Inside the if comparison clause, it repeats processes 1 and 2.

Let's imagine that this is the underlying query on this data frame:

df1 = spark.sql("SELECT * FROM catalog1.schema1.table1 where column1 = 'SPARK'")

The important thing is that if the data of catalog1.schema1.table1 is constantly being updated, it can return different results every time a Spark action is called (such as df.count()). This is the scenario when, for example, the table catalog1.schema1.table1 is being processed by a streaming. "But I'm not running a streaming and the counts sometimes don't match!". Well, your data is still being updated, so take a look at possible joins with tables that are being written over jobs or CDC processes.

And how do I check for the .count() statement evaluation inside SparkUI?

We are not covering here any tutorial on the SparkUI itself. I'm going to jump straight to the point on how to identify your counts. If you need to better understand the SparkUI, please refer to this link.

  1. The first step is to understand how many actions it will be triggered by your count. You can do this by analyzing how many other actions will be needed to compute your initial df.count(). Is your data frame related to a lot of different underlying tables? Do they have lots of joins, merges, and/or unions followed by grouping data? Or is it plain and simple, straight to the table itself? You don't need to know the exact amount but have an idea. You will use this "idea" when you're searching for queries/data frames on the last tab of the SparkUI, and relate them to the triggered jobs
  2. Were they the last thing to be triggered on your notebook/job? if so, did the df.count() command took a long time? This is especially useful in Databricks since you can separate these operations on specific commands and check the time it takes to run them on the top of the cell.

Now it's time to go to the UI -> SQL/Dataframe tab and search for your operation:

This next step might take a while for you to find, but if the .count() operations were intensive, they're probably the most lasting command, so you can filter by duration. If they aren't, you need to double-check the latest commands filtering by time and go from there.

That's why running those counts on a separate cell of your notebook is a good idea. You can check the exact job ID that ran for that specific query, and relate to it on the side, in the "Job IDs" column.

You can click the description of the command (highlighted in blue) to check the DAG for that Query/Dataframe.

Now you can directly see that the last WholeStageCodegen returns only one object (that count number) and that the Scan reads all the lines (since a .count() needs to read all the lines). There, you've finally found your number. This can change from plan to plan, but following from the top (that 1 row) downwards can help you navigate the transformations that were made to reach that single line, and get your number from there.

Optimizing your code

Besides all this .count() thing, running the above command is not the most optimized solution. I can understand that you need to compare, but you should save those results saved into two new variables (Scala users would hate it to be variables, tho).

The most optimized setting, in this case, is saving both and comparing after, like this:

count_df1 = df1.count()
count_df2 = df2.count()

if count_df1 == count_df2: # comparison happens with the variables, and does not trigger new actions
    print("count matches, proceeding further")
else:
    print("count does not match!")
